# CLAUDE.md — Луна Кофе (клиентская часть PWA)

Этот файл — главный источник правил для Claude Code в этом проекте. Читай его
первым в каждой сессии и сверяйся с ним перед каждым нетривиальным решением.

## Что мы строим

Клиентская часть PWA-приложения кофейни «Луна Кофе». Три пользовательские зоны:
**клиент**, **бариста**, **администратор**. Полный цикл: меню → конструктор
напитка → корзина → оформление → отслеживание статуса в реальном времени.
Интерфейс бариста управляет очередью заказов, админка показывает статистику и
экспорт CSV.

Это **только фронтенд**. Бэкенд появится позже (Node.js). Сейчас работаем на
**моках** (см. раздел «Работа с данными и моки»).

## Архитектура — Feature Sliced Design (СТРОГО)

Весь код в `src/`, разбит по слоям FSD. Подробные правила — в
`.claude/rules/fsd.md`. Кратко, порядок слоёв сверху вниз:

```
app → pages → widgets → features → entities → shared
```

> **Слой `pages` физически лежит в `src/views`** — каталог `src/pages`
> зарезервирован Next.js под Pages Router. Алиас `@pages/*` указывает на
> `src/views`. Импорты внутри слоёв — по сегментам (`@shared/ui`, `@shared/api`,
> `@entities/product` …), не из единого барреля слоя.

**Правило зависимостей:** слой импортирует только из НИЖЕЛЕЖАЩИХ. Импорт вверх
или вбок (feature из feature) — запрещён. Нарушение этого правила — стоп-сигнал,
переосмысли размещение кода.

Импорты между слоями — только через публичный API (`index.ts` слайса). Глубокие
импорты во внутренности чужого слайса запрещены.

## Что делать в начале каждой задачи

1. Прочитай `docs/TZ.md` — там зафиксированы требования. Не отклоняйся от них.
2. Сверься с `docs/ROADMAP.md` — там фазы и порядок (актуальный статус фаз — в
   блоке «Статус разработки» в начале файла). Делай текущую фазу, не убегай
   вперёд.
3. Сверься с `.claude/rules/fsd.md` — куда положить код по слоям.
4. Если задача затрагивает данные — посмотри `.claude/rules/mocks.md`.

## Definition of Done для любой задачи

- [ ] Код проходит `npm run type-check` без ошибок (strict TS).
- [ ] Код проходит `npm run lint` без ошибок и предупреждений.
- [ ] Соблюдены правила слоёв FSD и импортов через публичный API.
- [ ] Нет хардкода строк/чисел, которые должны быть в `shared/config`.
- [ ] Новые компоненты адаптивны (mobile-first, проверка на 375 / 768 / 1280).
- [ ] Нет прямых обращений к localStorage для токена (см. правила безопасности).
- [ ] Если добавлены данные — они идут через мок-слой, не зашиты в компонент.

## Стиль кода

- Функциональные компоненты, хуки. Классов нет.
- Именование файлов: kebab-case (`order-card.tsx`), компоненты — PascalCase.
- Один компонент — один файл. Логику выноси в `model/` слайса (хуки, стор).
- Типы — рядом с сущностью в `entities/*/model/types.ts` или в `shared/types`.
- Никаких `any`. Если тип неизвестен — `unknown` + сужение.
- Комментарии на русском, по делу. Не комментируй очевидное.

## Чего НЕ делать

- Не трогай бэкенд — его нет. Все серверные вызовы идут в мок-слой.
- Не пиши тесты, пока не дойдём до фазы тестирования в роадмапе.
- Не оптимизируй преждевременно. Сначала рабочий код по ТЗ.
- Не придумывай фичи сверх ТЗ. Если идея хорошая — предложи, но не внедряй сам.
- Не меняй структуру слоёв FSD под «как удобнее».
