# ROADMAP — порядок разработки по фазам

Claude Code идёт по фазам последовательно. Не перепрыгивай вперёд. Внутри фазы
можно выбирать порядок задач, но не выходи за её рамки без отметки о завершении.

В конце каждой фазы: прогон `type-check` + `lint`, краткий отчёт что сделано,
что осталось. Затем переход к следующей фазе.

---

## Статус разработки

Обновляется по мере прохождения фаз — быстрый ориентир для новой сессии.

- ✅ **Фаза 0** — каркас (Next.js 16, Tailwind 4, Shadcn/UI, FSD, линтинг)
- ✅ **Фаза 1** — слой shared (config, api+перехватчики, ui, lib, store, моки)
- ✅ **Фаза 2** — entities (product, order, user) на моках
- ✅ **Фаза 3** — навигация, route groups, RoleGuard, заглушки экранов
- ✅ **Фаза 4** — авторизация (формы входа/регистрации, сессия, 401, редирект по роли)
- ✅ **Фаза 5** — клиентский сценарий заказа (каталог, конструктор, корзина, оформление, статус)
- ✅ **Фаза 6** — реал-тайм (мок-сокет + BroadcastChannel) и очередь бариста
- ✅ **Фаза 7** — админка: статистика за период, популярные позиции, экспорт CSV
- ✅ **Фаза 8** — PWA: Serwist (преемник next-pwa), manifest, service worker, offline-страница
- ✅ **Корректировка (пропущено по ТЗ)** — личный кабинет клиента (профиль +
  редактирование + история заказов + выход), аккаунт-экран и навигация бариста/
  админа, выход на мобильных. Закрыто 2026-06-17, см. блок «⚠️ Пропущено по ТЗ».
- ⏭️ **Фаза 9** — полировка и визуальная доводка под Figma — следующая
- ⬜ Фаза 10 — впереди

> **PWA-движок:** `next-pwa` (shadowwalker) заброшен в 2022 и несовместим с
> Next 16 — взят его поддерживаемый преемник **Serwist** (`@serwist/next`, тот же
> автор, Workbox). Обоснование зафиксировано в `CLAUDE.md`. Иконки PWA, favicon и
> кофейные изображения берутся из импортированных ассетов (`assets/` → `public/`),
> не генерируются.

🎨 **Визуал:** фирменная зелёная тема (OKLCH, hue ~158) настроена в
`globals.css` — весь интерфейс в бренде. Попиксельная доводка экранов под Figma
и точечные анимации на **Motion (Framer)** из React Bits (вариант TS+Tailwind,
копируется в код, `prefers-reduced-motion`, без тяжёлых WebGL/3D — three/ogl/GSAP)
запланированы отдельной визуальной доводкой в Фазе 9.

Все изменения закоммичены и запушены в `origin/main`.

## Замечания код-ревью (тех-долг, по убыванию важности)

Найдено при ревью после Фазы 5. Распределено по фазам; устранять в порядке
важности. Отмечать сделанное.

- [x] **[Фаза 6] Цена быстрого «+» в каталоге ≠ цене на карточке.** `+` добавляет
      дефолтный набор, а дефолтный размер — первый из списка (S, −30₽), тогда как
      карточка показывает `basePrice` (= размер M). Починить вместе с дефолтом = M
      в `getDefaultSelection` (`entities/product/lib/configurator.ts`). Самое
      заметное пользователю.
- [x] **[Фаза 6] Hydration-mismatch в форме оформления.** `Date.now()` в
      инициализаторе `useState` пререндерится со временем сборки, а на клиенте — со
      временем рантайма (`features/place-order/ui/checkout-form.tsx`). Считать
      дефолтное время в `useEffect`/`setValue`, не в рендере.
- [x] **[Фаза 6] `useOrder` не сбрасывает состояние при смене `id`** — покажет
      старый заказ при переходе order→order (`widgets/order-status/model/use-order.ts`).
      Уйдёт при переработке экрана статуса под WebSocket.
- [x] **[Фаза 6] Нет кнопки выхода в UI.** `useLogout` реализован, но не подключён —
      добавить «Выйти» в навбар (особенно для зон staff/admin).

Из ревью Фазы 6 (реал-тайм) — обязательны к устранению на Фазе 7:

- [x] **[Фаза 7] Мок мутирует общие объекты заказа.** `updateOrderStatus` делает
      `order.status = status` прямо на элементе `MOCK_ORDERS`, а `getOrders`
      отдаёт `[...MOCK_ORDERS]` (копия массива, общие ссылки на объекты).
      Возвращать новый объект, а не мутировать (`entities/order/api/order.api.ts`).
- [x] **[Фаза 7] Непроверенный каст `payload as Order` из BroadcastChannel.**
      Проверять форму payload (есть `id`/`number`/`items`) перед добавлением в
      очередь — иначе чужой/битый `postMessage` уронит `OrderCard`
      (`widgets/orders-queue/model/use-orders-queue.ts`).
- [x] **[Фаза 7] Таймер авто-прогрессии не гасится при уходе клиента с экрана.**
      `useOrderSocket` отписывается, но `startMockProgression` докручивает до
      `ready`, дёргая колбэки без подписчиков. Останавливать прогрессию на
      анмаунте (`shared/realtime/order-hub.ts`, `use-order-socket.ts`).
- [x] **[Фаза 7] `JSON.parse` в `onmessage` реального WS без try/catch.** Битый
      фрейм с бэка кинет исключение в обработчике (`shared/realtime/use-order-socket.ts`).

Из ревью Фазы 7 (админка) — обязательны к устранению на Фазе 8:

- [x] **[Фаза 8] Неограниченный период статистики.** `getOrderStats` создаёт
      бакет на каждый день диапазона, а `SalesChart` — бар на каждый день; date-
      инпуты принимают любые даты, поэтому период в годы → тысячи бакетов/DOM-узлов
      и риск `Math.max(...spread)`. Ограничить размах периода (≤ ~366 дней) в
      `getOrderStats`/UI (`entities/order/api/order.api.ts`, `widgets/stats-dashboard`).
- [x] **[Фаза 8] `generateOrderHistory` пересоздаётся на каждый запрос.** ~840
      заказов генерируются заново при каждой смене периода — мемоизировать по дню-
      якорю (`entities/order/api/order.api.ts`, `stats.mock.ts`).
- [x] **[Фаза 8] `ExportCsvButton`: ссылка не добавлена в DOM перед `click()`.**
      Часть браузеров требует `<a>` в документе для скачивания — append+remove для
      совместимости (`features/export-csv/ui/export-csv-button.tsx`).
- [x] **[Фаза 8] `toCsv`: одиночный `\r` не экранируется.** Значение с возвратом
      каретки без `\n` может разорвать строку CSV — добавить `\r` в проверку
      (`features/export-csv/lib/to-csv.ts`).
- [x] **[Фаза 8] `isOrder` не проверяет форму элементов `items`.** Payload с
      `items: [мусор]` пройдёт guard и уронит `OrderCard` при рендере — проверять
      хотя бы ключевые поля позиций (`entities/order/model/types.ts`).
