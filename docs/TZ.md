# ТЗ — Клиентская часть PWA-приложения кофейни «Луна Кофе»

Источник истины по требованиям. Claude Code не отклоняется от этого документа.
Если требование непонятно или конфликтует с другим — спроси, не домысливай.

## 1. Цель

Адаптивное клиентское PWA-приложение кофейни: просмотр меню, конструктор
напитка, оформление заказов, управление профилем, отслеживание статуса заказа.
Дополнительно — интерфейс бариста и административный раздел.

## 2. Пользовательские роли

- **Клиент** — гость или авторизованный. Полный цикл заказа.
- **Бариста** — сотрудник. Очередь заказов, смена статусов.
- **Администратор** — статистика, экспорт CSV.

Разграничение доступа — на уровне маршрутов через проверку роли в токене.

## 3. Функциональные требования

### 3.1 Просмотр меню
- Категории товаров.
- Поиск и фильтрация позиций.
- Карточка товара: изображение, описание, стоимость.

### 3.2 Оформление заказа
- Корзина: добавление, изменение количества, удаление.
- Оформление заказа.
- Выбор времени получения (не ранее чем через 10 минут от текущего момента).

### 3.3 Конструктор напитка
- Выбор размера.
- Выбор типа молока.
- Дополнительные ингредиенты.
- Стоимость пересчитывается автоматически при изменении параметров.

### 3.4 Личный кабинет
- Вход и регистрация.
- История заказов.
- Текущий статус заказа.
- Редактирование профиля.

### 3.5 Интерфейс сотрудника (бариста)
- Просмотр входящих заказов в реальном времени.
- Изменение статуса заказа: принят → готовится → готов.
- Отображение текущей очереди.

### 3.6 Административный раздел
- Статистика заказов за выбранный период.
- Наиболее популярные товары.
- Данные о продажах за период.
- Экспорт данных в CSV (на стороне клиента, без сервера).

### 3.7 PWA
- Установка на устройство.
- manifest-файл.
- Базовое кеширование статических ресурсов через service worker.
- Offline-страница при отсутствии сети.

## 4. Нефункциональные требования

- Адаптивность: мобильные и десктоп, подход mobile-first.
- Брейкпоинты для проверки: 375 / 768 / 1280 px.
- Навигация: на мобильных — нижняя панель (bottom nav), на десктопе — верхний
  хедер с горизонтальной навигацией. Переключение средствами CSS, без условного
  рендеринга на JS.
- Стабильная работа в актуальных версиях браузеров.
- Строгая типизация TypeScript (strict).
- Компонентная архитектура.
- Базовая валидация пользовательских данных (Zod).
- Разграничение доступа по ролям.

## 5. Требования к безопасности

- Корректная работа с токенами авторизации.
- НЕ хранить чувствительные данные в небезопасных источниках. Токен — НЕ в
  localStorage напрямую (уязвимо к XSS). Хранить в памяти (Zustand store);
  сброс при перезагрузке — приемлемый компромисс на этапе фронтенда без бэка.
- Корректная обработка ошибок сети (централизованно через перехватчики Axios).

## 6. Основные экраны

Клиентская зона:
- Главная страница
- Меню (каталог)
- Карточка товара (с конструктором)
- Корзина
- Оформление заказа
- Статус заказа
- Личный кабинет
- Авторизация / Регистрация

Зона бариста:
- Очередь заказов

Админ-зона:
- Дашборд статистики (+ экспорт CSV)

## 7. Что вне scope текущего этапа

- Реальный бэкенд (будет на Node.js позже). Сейчас — моки.
- Реальная онлайн-оплата (только заглушка интерфейса, если потребуется).
- Push-уведомления (перспектива, не сейчас).
- Программа лояльности (перспектива).
- Автоматизированные тесты (отдельная фаза в конце).
- Мультиязычность.

## 8. Критерии готовности проекта

- Реализованы основные пользовательские сценарии всех трёх ролей.
- Приложение корректно работает на мобильных и десктопных устройствах.
- Реализована авторизация и разграничение по ролям.
- Функционирует оформление и обработка заказов (на моках).
- PWA: манифест, service worker, установка, offline.
- Подготовлена документация по запуску.
