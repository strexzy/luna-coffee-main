# Правила работы с данными и моками

Бэкенда сейчас нет. Он появится позже (Node.js). Задача — сделать фронт так,
чтобы переход на реальный API был бесшовным: поменять только реализацию внутри
api-слоя, не трогая компоненты.

## Главный принцип

Компоненты и хуки НЕ знают, что данные мокованные. Они вызывают те же
API-функции (`getProducts`, `createOrder`...), что будут работать с реальным
бэком. Мок живёт ВНУТРИ этих функций или за переключателем.

```ts
// entities/product/api/product.api.ts
export const getProducts = async (category?: string): Promise<Product[]> => {
  if (USE_MOCKS) {
    return mockGetProducts(category); // имитация задержки + мок-данные
  }
  const { data } = await apiInstance.get<Product[]>('/products', {
    params: { category },
  });
  return data;
};
```

`USE_MOCKS` — флаг из `shared/config`, читается из переменной окружения
(`NEXT_PUBLIC_USE_MOCKS`). По умолчанию сейчас `true`.

## Где лежат моки

Решение зафиксировано 2026-06-12: мок-данные сущностей живут РЯДОМ с их типами
и api-функциями в `entities/*/api/`, иначе shared пришлось бы импортировать
типы из entities — импорт вверх, запрещённый правилом слоёв FSD.

```
shared/api/mocks/           # только домено-независимая инфраструктура
├── mock-helpers.ts         # withDelay, maybeFail — имитация сети
└── index.ts

entities/product/api/       # данные + api-функции сущности (фаза 2)
├── product.api.ts          # if (USE_MOCKS) → мок, иначе axios
└── product.mock.ts         # мок-каталог
```

Мок-функции имитируют сеть: небольшая задержка (200–500мс через setTimeout/
Promise) и иногда — ошибки, чтобы можно было проверить обработку. Хелперы
задержки/ошибок — общие, из `shared/api/mocks`.

## Контракты типов — единые

Тип ответа мока === тип будущего ответа бэка. Описывается ОДИН раз в
`entities/*/model/types.ts`. Когда появится бэк, типы не меняются — меняется
только тело api-функции. Это и есть бесшовный переход.

## Моки авторизации

Три тестовых пользователя — клиент, бариста, админ. Логин по любому из них
возвращает мок-токен и роль. Перехватчик и редиректы работают как с реальным
API.

Auth-сеам реализован полностью в `entities/user/api`: `login` / `register` /
`refresh` / `logout` / `getProfile`. В моке `refresh` отдаёт 401 — без
httpOnly-cookie и бэка сессия не переживает перезагрузку (целевая схема —
TZ §5). Мок-токен кодирует id пользователя (`mock-token.<id>`).

```ts
// примеры тестовых логинов задокументируй в README
client@luna.test  / 12345678  → роль client
barista@luna.test / 12345678  → роль staff
admin@luna.test   / 12345678  → роль admin
```

## Мок WebSocket

Реального сокета нет. Хук `useOrderSocket` за флагом `USE_MOCKS` использует
мок-эмулятор: по таймеру меняет статус заказа (принят → готовится → готов),
вызывая те же колбэки, что и реальный сокет. Интерфейс хука НЕ зависит от того,
мок это или реальный WebSocket.

## Ассеты из Figma

Студент импортирует ассеты (иконки, изображения) из макета Figma вручную.
Claude Code:

- кладёт статику в `public/` (иконки PWA, изображения товаров-моков);
- НЕ генерирует «заглушечные» картинки сам, если студент сказал, что импортирует
  из Figma — вместо этого оставляет понятные плейсхолдеры и помечает `// TODO:
заменить на ассет из Figma`.

## Что нельзя

- Зашивать мок-данные прямо в компонент (только через api-слой).
- Делать так, что компонент знает про моки (никаких `if (mock)` в UI).
- Ломать контракт типов ради удобства мока.
